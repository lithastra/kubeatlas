## 2. P5-T9 perf baseline — ✅ DONE (both tiers captured live)

`test/verify/perf-baseline-v1.5.json` is committed, measured on a live
Docker Desktop cluster (k8s v1.36.1, WSL2): Tier 1 = the memory-backed
v1.5 binary; Tier 2 = the same binary against a standalone
`apache/age:release_PG16_1.6.0` Postgres+AGE pod. GraphStore v2 shows no
regression — both tiers meet every SLO with large headroom (Tier 2
cluster/namespace p95 ~29/39ms vs the 1000ms budget, blast p95 ~4ms vs
500ms), and `graphstore_version` reports `"v2"`.

This run also validated the **whole OTel pipeline end-to-end** against
real Postgres+AGE (not just unit tests): migration 010 applied
(`otel_runtime_edges` + `otel_spans` tables present); a two-span trace
sent to the `:4317` receiver was stored, and the background correlator
resolved both services to real graph Deployments (`0` unmatched) and
produced the `CALLS_AT_RUNTIME` edge, which `/api/v1/otel/overlay`
returned and `compare=true` classified `observed_only`.

Caveat: the baseline used the **1k** fixture (deliberately light for a
local dev cluster), not the v1.4 10k fixture — so it proves SLO
compliance and no v2-store regression, not a same-magnitude comparison
to v1.4. **A 10k re-run belongs in CI / the release environment:**

```bash
bash test/perf/stress-10k-resources.sh
# Tier 1: run the memory binary locally, then:
KUBEATLAS_TIER=tier1 NS=stress-test-10k \
  KUBEATLAS_BASELINE_OUT=test/verify/perf-baseline-v1.5.json \
  KUBEATLAS_BASELINE_PHASE=5-v1.5.0 bash test/perf/bench-v1.sh
# Tier 2: repeat with the chart-installed CNPG server, KUBEATLAS_TIER=tier2.
```

Gate: v2 store ≤ 5% regression vs v1; OTel correlation is a background
job (≤1min cadence, off the request path — not a request-latency SLO).

## 7a. Pre-existing follow-up (not a v1.5 blocker) — stale api-compat baseline

`api/openapi-v1alpha1.json` is a Phase-2 floor (16 paths). The live
v1alpha1 surface has grown backward-compatibly since (23 paths — labels,
networkpolicy, snapshots, export — plus a required `CycleReport.category`
field), all from v1.1–v1.4, none from v1.5. Because
`.github/workflows/api-compat.yml` only triggers on changes to that file,
the drift went unnoticed. It does not affect v1.5 (Phase 5 added nothing
to v1alpha1, proven by the live gate), but the baseline should be
refreshed so future compat checks are meaningful:

```bash
go run ./tools/api-compat-check/dump > api/openapi-v1alpha1.json
git add api/openapi-v1alpha1.json && git commit -s   # reviewed: additions only, no removals
```

Recommend also widening the api-compat.yml trigger so the check runs on
every API change, not only when the baseline file itself changes.
